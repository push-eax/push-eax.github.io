Web development, minimalism, and the value of starting from scratch

by Kiernan Roche on March 19 2021

I have a habit of building web stuff (sites, themes, applications) from scratch,1 which takes longer than tweaking something off-the-shelf (e.g. WordPress), but allows me to better express my design philosophy. In recent years, that philosophy has converged on simplicity and minimalism. Here's why:

Starting from someone else's code means fixing problems they unknowingly create for you

When working on personal websites early on, instead of designing and implementing complex features, I would sometimes throw up my hands, copy an existing open source project (e.g. a CMS) that was close enough to what I was trying to do, and modify it to suit my needs. This practice is fairly widespread due to the growth of Q&A and tutorial sites, with many programmers copying code from the web and pasting it into their project without much consideration for how the code works, or in some cases using an open source project as a base, as I did.

While this is a perfectly rational thing to do in the interest of saving time, its cost is that you don't fully understand the code and all its quirks and behaviors, which is risky. Those quirks might not have been a problem for the person originally writing the code, but they may become a problem for you, because you need to modify the code to suit your requirements. Because you haven't taken the time to grok the codebase you just copied, you struggle to diagnose and properly fix any issues that arise, and development becomes an expensive game of whack-a-mole at best, or a complete rewrite at worst, which is what you were trying to avoid in the first place.

Building from scratch avoids this problem by making you aware of potential issues as you build them instead of when they occur, since you understand the code you write yourself. With good design, this reduces the time cost of maintenance and feature development.

Bloat is a usability and aesthetic problem

As artfully described here, modern websites are bloated. As a user, I'm just looking for information, unless I'm using a web application, in which case snappy load times and responsiveness are even more important. Megabytes of assets and JavaScript libraries that add several seconds of load time for the sake of smooth transition animations impede my search for information, which is an annoyance at best and will drive me elsewhere at worst.

While not strictly causal, bloated web frameworks seem to go hand-in-hand with loud and annoying UX. The sites that scream and poke and beg me to sign up for their newsletter or incessantly throw advertisements in my face also seem to be the ones that load most slothfully.

Part of this phenomenon is the unnecessary inclusion of heavy libraries like jQuery to accomplish things that can easily be done with a snippet of JavaScript or even pure CSS, but I suspect that one of the biggest reasons for this bloating is that, in an attempt to improve backend performance, web developers have offloaded more of the application logic and page rendering work to browsers. This shift is the result of two primary factors:

  1. The tradeoff of space for time that modern browsers make. RAM is plentiful nowadays, so browsers reserve more of it and use it to make browsing faster. This allows developers to keep massive amounts of data (assets) in the browser, and rely on the browser to share more of the workload. Developers take advantage of this, increasing the memory (and network and CPU) footprint of modern web apps, so browsers reserve even more memory, and the cycle continues.
  2. As a consequence of (1), the rise in popularity of single-page apps (SPAs) and the proliferation of JavaScript frameworks used to build SPAs (Angular, React) to web things that aren't SPAs. Not only does this sap browser performance, but it also makes certain kinds of security issues (e.g. DOM-based XSS) more common.

Building from scratch avoids bloat by implementing and including only the features that are necessary to fulfill the requirements, avoiding the inclusion of unused library code.

Complexity creates flaws

There are good reasons to avoid the blight of bloat aside from usability and appearance. It's difficult, maybe even impossible, for human beings to fully understand the behavior of complex systems in all cases, which means that predicting misbehavior is very difficult. We can minimize this by:

  • Making systems only as complex as they need to be to fulfill their requirements,2 and
  • choosing those requirements carefully so as to encourage a simple implementation (do one thing, or a very small set of closely related things, and do them well).

This site, for example, is statically generated and has no backend other than the web server itself, which eliminates any application-level vulnerabilities and improves load times dramatically. I could run this site on a CMS, but the complexity of such a solution is unnecessary if I'm not using any of the additional features, such as comments or a contact form. Conversely, I could choose to add support for comments or a contact form, but that would necessitate greater implementation complexity. Minimizing requirements makes it easy to minimize complexity, and vice versa.

Don't reinvent the wheel

It's easy to make decisions about requirements for web software when browsers already have the tools necessary to perform most functions that web developers re-implement:

  • Browsers have built-in search. Try it out: go to my blog archive, hit Ctrl+F,3 and search for a blog post. Failing that, my site is also indexed by search engines, so you can do stuff like this. Unless I need a more powerful or flexible search function, why would I implement that again? The browser's implementation is faster than anything I could write and creates no application-level vulnerabilities or undesirable behavior.
  • Browsers have mailto: links. Why do I need a contact form when I have a tiny link that automatically opens a user's mail client with the To and Subject fields already filled out? The biggest reason I can think of is spam, but mailto: links can be obfuscated and hidden behind a CAPTCHA with a little JavaScript, and unlike contact forms, they require no backend to process POST requests. On this site, I chose to use some HTML tricks to obfuscate my email address rather than JavaScript, so you, my poor reader, must copy and paste my address into your mail client, or type it. Terrible.
  • Browsers have forward and back buttons for navigation, another thing broken by SPAs. Now we need "routers", which are JavaScript libraries that re-implement navigation, which was working perfectly well already. As usual, technological progress solves problems and creates new ones.

Subjectivity

Function-over-form is more than just a way to design and build websites or software. It's a way of life for me, and it informs every decision I make. Ergo, this site's minimalism has a greater purpose than pleasantness of appearance; it's an expression of my own personality, aesthetic, and artistic sensibility, which is unique to me and may not be appreciated by everyone. I'm sure there are those who, out of some strange masochism, enjoy being overwhelmed by useless and redundant things completely extraneous to the experience of searching for information.

I choose to use the tools we have available to us as web developers sparingly, deliberately, and precisely. I encourage you to use them that way as well.

Here are some sites I'd like to commend for their minimalism:

  • Tarsnap, though one could argue that this is a specific design flavor (Unix-chic?) that is not completely described by the word "simple".
  • prgmr, which offers not only a minimalistic website but also a control panel over ssh.
  • Anything by Drew DeVault, who talks about the bloat issue in more detail here.
